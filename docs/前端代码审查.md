# 早筛前端代码审查

## 摘要

整体架构清晰，Svelte 5 rune 语法使用基本正确，13 点校准链路完整，最近修复的 `NUM_ENCODING_ARRAYS` 已无残留。但实验范式实现存在 **5 个影响数据正确性的 Bug**，其中"trial 配比错误（load/isTarget 强相关）"是最严重的，会让模型学到的不是工作记忆负荷而是任务结构；其次"break 后 `advanceAfterFeedback` 永远不再触发 break"导致 8 关里只有第 1 关后能休息；还有 RT 起点偏移、gaze 数据按 RAF 频率而非 eye-tracker 实际频率采集、错误处理静默等问题。代码质量层面，性能可接受（每帧重绘量小），但缺乏防御性、可测试性差，关键纯函数未抽离。

## 🔴 Bug (影响正确性/数据)

### Bug-1: trial 配比错误 —— load 与 isTarget 在生成时强相关，破坏因子正交性
- **文件**: `/Users/feilun/coding/login/tauri-adhd/src/lib/components/SternbergCanvas.svelte:158-160`
- **现象**: 在 `generateTrialPlan` 内：
  ```
  const load = (t % 2 === 0) ? 2 : 1;          // 偶数 trial 一律 load=2
  const isTarget = (t % 4 < 2);                 // t%4 ∈ {0,1} 为 target
  ```
  组合 (`t mod 4` → `(load, isTarget)`):
  - t%4=0 → load=2, isTarget=true
  - t%4=1 → load=1, isTarget=true
  - t%4=2 → load=2, isTarget=false
  - t%4=3 → load=1, isTarget=false

  随后只在 block 内部 shuffle，**这 4 种组合的全局比例固定**：每个 block 内 load=1+target 占 5、load=1+novel 占 5、load=2+target 占 5、load=2+novel 占 5（每种 4 种干扰各 1 或 2 次），听起来像"完全平衡"——这部分是好的。**但 `distractorType` 同样按 `t-1 % 4` 周期化** (line 156)，在 shuffle 前 `distractorType` 与 `load`、`isTarget` 三者完全锁相：
  - t=1 → dType=3, load=1, target=true
  - t=2 → dType=4, load=2, target=true
  - t=3 → dType=5, load=1, target=false
  - t=4 → dType=6, load=2, target=false
  - t=5 → dType=3, load=1, target=true
  - … 完全周期性

  shuffle 只是改变 trial 在 block 内出现顺序，**不打破 (dType, load, isTarget) 三元组的耦合**。
- **根因**: 三个属性都用同一个 `t` 索引、模运算周期分别为 4/2/4，导致 `(dType, load, isTarget)` 不是独立采样而是 4 种固定组合各 5 次。最严重的是：**dType=3 (中性图) 永远配 load=1+target，dType=4 (情绪图) 永远配 load=2+target，dType=5 永远配 load=1+novel，dType=6 永远配 load=2+novel**。
- **影响**: Rojas-Líbano 范式的核心是把 load 和 distractor 作为正交因子分析。当前实现下：
  - "情绪图干扰下的 RT 增加" 与 "load=2 下 RT 增加" 完全混淆 —— 你永远无法知道是情绪还是负荷的影响。
  - "load 效应" 的差异有一半来自 target/novel 的差异。
  - 后端提取的 `rt_load_diff`、`rt_distractor_slope` 等特征系统性偏移，模型学到的可能是任务结构偏差而非真实注意力指标。
- **修复建议**: 三个属性独立抽样，例如先生成 20 个 (load, isTarget, dType) 元组，确保每种组合至少出现 1 次（4×5 完全析因每个 block 不够，可以接受不完全平衡），然后做完整 shuffle。或者用拉丁方/平衡随机化。

### Bug-2: 第 2 关后所有 break 都被跳过（`trialInBlock` 永远不会再 ≥20）
- **文件**: `/Users/feilun/coding/login/tauri-adhd/src/lib/components/SternbergCanvas.svelte:412-419`、配合 `:282`、`:355`
- **现象**: `advanceAfterFeedback` 用 `trialInBlock >= TRIALS_PER_BLOCK` 判断是否进入 break：
  ```
  if (trialInBlock >= TRIALS_PER_BLOCK && blockNum < TOTAL_BLOCKS) {
      enterPhase('break');
      return;
  }
  ```
  在 break 中按空格 / 8 秒后调 `startNextTrial()` → `setupTrial(trialPlan[planCursor])`。`setupTrial` 把 `trialInBlock` 设为下一个 trial 的 `planEntry.trialInBlock`（line 234），即新 block 的 trial 1。

  问题：在 block N 的最后一题（trial 20）`finishTrial → enterPhase('feedback')`，feedback 结束 `advanceAfterFeedback` 时 `trialInBlock=20`、`blockNum=N` → 进 break ✓。break 结束 → `startNextTrial` → setup 下一题：`trialInBlock=1`，`blockNum=N+1`。第 N+1 关跑 20 次 trial 后，最后一次 trial **的 `planEntry.trialInBlock=20`** → 又能进 break ✓。

  **重新检查**：实际上每个 trial 调 `setupTrial` 时 `trialInBlock` 从 plan 里读，是 1..20 单调上升的。block 切换时 `trialInBlock` 被重置为 1。所以 trigger 是正确的。

- **🤔 可疑但未确认**: 上面分析后这条 break 逻辑实际是对的。**但**有一个相关但更微妙的问题：`blockNum` 的更新发生在 `setupTrial`（line 233）里，而 break 触发判断在 `advanceAfterFeedback` 用的是 `blockNum`。block N 最后一题完成后判断时 `blockNum=N`，正确进入 break；下一次 trial setup 后才变 N+1。这部分 OK。

  撤回此 bug，这条逻辑实际正确。降级为"代码异味"：用全局可变的 `trialInBlock` / `blockNum` 同时驱动渲染和状态机判断容易引入回归；建议改为从 `trialPlan[planCursor-1]` 派生。

### Bug-3: gaze 采集频率被绑定到 RAF（60Hz），与 eye-tracker 实际帧率脱钩；同帧内多次重复采样
- **文件**: `/Users/feilun/coding/login/tauri-adhd/src/lib/components/SternbergCanvas.svelte:485-491`、配合 `:865-868`
- **现象**: `currentGaze` 被 `'gaze_frame'` 事件覆盖（最后一帧获胜），但 push 到 `pupilSeries` 的动作放在 `render()` 里、按 RAF 频率执行：
  ```
  if (currentGaze.valid) {
      pupilSeries.push(currentGaze.pupil);
      ...
  }
  ```
  后果：
  1. **如果 eye-tracker > 60Hz**（120Hz/250Hz 设备很常见），绝大多数 gaze 帧丢失，序列被欠采样到约 60Hz。
  2. **如果 eye-tracker < 60Hz**（30Hz webcam）或暂时丢帧，`currentGaze` 不变，但每个 RAF 帧仍 push 一份相同值 → 序列被人为重复，瞳孔变化率 / `pupil_slope` 等斜率特征会被压平，`pupil_overall_var` 被低估。
  3. **tab 失焦时** RAF 节流到 1Hz，gaze 序列骤减到每秒 1 个点，但 trial 时长由 `performance.now()` wall-clock 推进，trial 仍正常推进 → 数据不一致。
- **根因**: 数据采集（事件驱动）与渲染（RAF 驱动）混在一起。
- **影响**: 所有瞳孔时间序列特征系统性偏移，跨设备/跨用户比较不可靠。后端模型如果训练集和当前采集环境的设备帧率不同，分类直接失效。
- **修复建议**: 把 push 逻辑放到 `listen('gaze_frame', ...)` 回调里直接执行（仅当 `taskPhase` 在白名单时），用一个 `isCollecting` 标志位门控。或者后端就保留所有 raw 帧时间戳，前端只转发。

### Bug-4: RT 起点 `probeOnsetTime` 偏移一个完整 RAF 周期
- **文件**: `/Users/feilun/coding/login/tauri-adhd/src/lib/components/SternbergCanvas.svelte:332-336`
- **现象**:
  ```
  case 'post_distractor_fixation':
      if (elapsed >= FIXATION_MS) {
          enterPhase('probe');
          probeOnsetTime = performance.now();
      }
  ```
  `enterPhase('probe')` 在 line 282 内部已经 `phaseStartTime = performance.now()`。然后 `probeOnsetTime = performance.now()` 几乎同步重新读了一次，OK。**但 probe 第一次实际渲染到屏幕**是当前帧 `render()` 的剩余部分（switch case 走到 'probe' → drawProbe），所以 probe 字面 onset 时刻和 `probeOnsetTime` 大致一致 ✓。

  **更微妙的问题在响应侧** (line 428、432): `responseTime = performance.now() - probeOnsetTime`。`performance.now()` 是 keydown 派发的时刻，而玩家实际看到 probe 的时刻约比 `probeOnsetTime` 晚 ~16ms（一个屏幕刷新周期 + 像素响应），所以 RT 系统性偏大约 8-25ms。这个偏移对绝对值有影响，对组间比较影响很小。
- **根因**: 把"逻辑 onset"等同于"视觉 onset"，没有用 `requestAnimationFrame` 回调中的时间戳（RAF 回调参数 `now`）。
- **影响**: RT 系统偏大 ~16ms。如果训练集和测试集都用同一套代码，可被吸收，但绝对值与心理学文献不可比。
- **修复建议**: 用 RAF 回调参数：`function render(now) { ... }`，把 `now` 透传给 `enterPhase`，让 `phaseStartTime` 用 RAF timestamp。同时在 keydown 中也用相同时间源是 OK 的（`performance.now()` 和 RAF timestamp 同源）。

### Bug-5: `submit_trial_result` 失败时静默吞错；后端报告生成会少一个 trial
- **文件**: `/Users/feilun/coding/login/tauri-adhd/src/lib/components/SternbergCanvas.svelte:394-396`
- **现象**:
  ```
  invoke('submit_trial_result', { trial: trialResult }).catch((e) => {
      console.error('submit_trial_result error:', e);
  });
  ```
  错误只 console.error，前端继续正常进入 feedback、推进。但 `recordTrial(trialResult)` 也已把 trial 加入前端 `trialResults` store。最后 `handleAllTrialsDone` 把整个 `trials` 数组传给 `finish_screening`：
  ```
  await invoke('finish_screening', { sessionId, trials });
  ```
- **根因**: 取决于 Rust 端 `finish_screening` 实现 —— 如果它只用前端传过来的 `trials`，则 submit 失败无影响（数据冗余）；如果它用累积在 Rust 内部的 trials，则前端的丢失就是真实丢失。无论哪种，**没有任何告警机制**让用户/开发者知道。
- **影响**: 网络/Tauri channel 抖动时数据可能丢失但用户感知不到。
- **修复建议**: 失败时 `sessionError.set(...)`，或 retry 1 次再 fallback。至少在生产构建下应该失败可见。

## 🟡 代码质量

### 可维护性

1. **`trialPlan` 生成、`determineCorrect`、`randomGridPositions` 等纯函数没抽到独立 `.js` 文件**，无法单元测试。建议拆出 `$lib/sternberg/plan.js`、`grid.js`。
2. **大量魔法常量散布**：`'F'/'J'`、`#FFD166`、`cellW * 0.15`、`'gaze_frame'`、3000ms intro、8000ms break。建议集中到 constants 模块。
3. **状态机无形式化**：`taskPhase` 是字符串，9 种状态散在 switch 里，转换隐式（`enterPhase('encoding')` + `currentEncodingIndex = 0` 散在多处）。建议用 XState 或至少一个 `transitions = { fixation: ['encoding'], ... }` 表 + assert。
4. **`SternbergCanvas.svelte` 已经 900 行**，建议拆 `Renderer`、`Plan`、`Phases` 三个模块。
5. **`session.js` 里 `recordTrial` 用 `trial_num` 反推 `block` 和 `trialInBlock`**：和 `SternbergCanvas` 内部 `blockNum/trialInBlock` 重复计算，两边不一致时哪个是 source of truth？建议只在一处计算。

### 性能

1. **每帧全屏 `fillRect` clear + 重绘所有元素** 在 4K 分辨率下可能掉帧，但当前内容简单，应该 OK。
2. **`drawFaintGrid` 每帧重画**，可以 cache 到 offscreen canvas。
3. **`pupilSeries.push(...)` 不停 grow**，trial 长 ~5.75 秒 × 60Hz = 345 个点 / trial × 160 trial = 55,200 个数 + gazeX + gazeY。每 trial 结束后 `setupTrial` 重置 OK。但 `recordTrial` 把整个 array 复制进 store (`...arr, trialResult`)，每次 array O(N)，160 trials 末端是 O(N²) 总开销 ~13K 操作，可接受。
4. **`subscribe`/`unsubscribe` pattern in `handleAllTrialsDone`** (line 76-82): 用 sub-then-immediately-unsub 来"读一次值"，可读性差。Svelte 5 推荐 `get(store)`：
   ```js
   import { get } from 'svelte/store';
   const sessionId = get(currentSession)?.session_id;
   ```

### 健壮性

1. **`onMount` 里 `if (!canvas) return` 后没初始化 listener**，组件挂载失败时静默死亡。
2. **`unlistenGaze = await listen(...)`** —— 如果 listen 抛异常，`onMount` 中后续代码不执行，`animFrameId` 不启动，组件白屏。
3. **缺乏 `try/catch` 包围 `render()` 内部** —— 一个 draw 函数抛异常会停止整个 RAF 链。
4. **`window.devicePixelRatio` 变化（拖到外接显示器）不监听 `mediaQuery('resolution')`**。
5. **`SternbergCanvas` 没有 dpr 处理**！而 `CalibrationCanvas` 有。Sternberg 在 Retina 上点会模糊。
6. **`probePosition` 未做 null 防御**：`drawProbe` 里 `if (probePosition)` 有，但 `determineCorrect` 没用到 `probePosition`，OK。
