# Rust 后端眼动管线审查

> 审查范围: `tauri-adhd/src-tauri/src/{camera,face_mesh,pipeline,gaze_math,commands}.rs`
> 参照实现: `engine/adhd_engine/worker/gaze_system.py` (Python 基线)
> 前端契约: `tauri-adhd/src/lib/components/{CalibrationCanvas,SternbergCanvas}.svelte`

## 摘要

整体架构是清晰的: 摄像头线程 → latest_frame Mutex → 管线线程拉取 → emit `gaze_frame`。
线程模型用 `parking_lot::Mutex` 而非 std 的 Mutex, 已经规避了锁投毒问题;
用独立线程驱动 `!Send` 的 `nokhwa::Camera` 也是正确的 Rust 写法。然而:

- **特征契约与 Python 基线不一致** — 3 处会让 Ridge 回归器拟合到"不同口味"的特征
  (`LEFT_EYE_INDICES` 定义变了、eye 裁剪 margin 单位变了、rect_feature 用 xyxy 差
  算宽高而没 clamp max(1, ...))。
- **校准数据管道破碎** — 前端把 `[x, y, pupil, 0, 0, 0, 0, 0]` 当作 8D 特征
  提交 (CalibrationCanvas.svelte:72), 而不是后端 pipeline 当前的 8D 特征, 所以
  `submit_calibration_sample` 收到的 X 根本不是 `finish_calibration` 应该回归的那个空间。
- **屏幕坐标尺度不匹配** — `GazePipeline::new(..., 1920, 1080)` 写死,
  但前端校准点在 `window.innerWidth/innerHeight`。Ridge 学到 y∈[0,h_window],
  `process_frame` 内部再按 `clamp(0, 1920)` clamp。
- **AFFNet 重依赖** — 虽然有 graceful fallback (`None`), 但 8D 特征的第 7/8 维
  只要 AFFNet 没加载就恒为 `[0.0, 0.0]`, Ridge 训练时第 7/8 维会被 StandardScaler
  的 `std=1e-10` 兜底, 可能把噪声放大成爆表值。

### 评分速览
- 并发安全: 4/5
- 管线正确性: 2/5  (跨语言特征不对齐 + 屏幕尺度写死 + 前端给的校准特征都是零)
- 错误恢复: 3/5
- 性能: 2/5  (每帧 ~3 份完整 RGB 拷贝 + 多次像素级 for 循环填 tensor)
- 代码质量: 3/5

## 🔴 Bug (会让功能错或崩溃)

### B1. 校准特征通道错位 — Ridge 拟合到全零输入
**文件**: `commands.rs:210-229`, `CalibrationCanvas.svelte:70-73, 126-130`

前端在 `CalibrationCanvas.svelte:70-73` 这样填 features:
```js
const { x, y, pupil } = event.payload;           // x, y 已是屏幕像素
gazeFeatures = [x, y, pupil, 0, 0, 0, 0, 0];     // 后 5 维恒为 0
```
然后 `submit_calibration_sample(features=[...])` 把这个 8D 原封不动存到
`state.calibration_samples`。后端 `finish_calibration()` (commands.rs:242-247)
直接拿这些 X 去 `pipe.calibrate()`。问题:

1. 这个 8D 里第 0/1 维是**屏幕像素 x/y** (`gaze_frame` 已经是 ridge.predict 的输出),
   第 2 维是 pupil, 后 5 维恒 0。而 `pipeline.rs:289-298` 的 8D 定义是
   `[iris_rx, iris_ry, iris_lx, iris_ly, head_yaw, head_pitch, model_gx, model_gy]`。
   训练数据是 (screen_xy, pupil, 0, 0, 0, 0, 0), 预测时输入是
   (iris, head, model_gaze) — **训练与预测完全不同空间**, Ridge 权重全无意义。
2. `pipeline.rs:427-429` 已经存在一个 `current_feature()` 方法正确返回后端 8D,
   但 calibration 完全没调它。

**修复建议**: `submit_calibration_sample` 应当 *不接收* features, 改为
后端从 `state.pipeline.current_feature()` 自行采样; 或新增
`get_current_feature` 命令供前端使用。

### B2. Pipeline 构造时屏幕尺寸写死 1920x1080 — 导致 Ridge 输出被错误 clamp
**文件**: `commands.rs:105`, `pipeline.rs:245-319`

`start_screening()` 里:
```rust
pipeline::GazePipeline::new(&state.models_dir, 1920, 1080)   // :105
```
但前端 `CalibrationCanvas.svelte:93-94, 118-119` 给的
`screenX/screenY = targetX/targetY = currentPoint.x * cssWidth` 是
**当前窗口像素**。在 4K / 2K / 1440p 显示器上, 若窗口 > 1920x1080, 则:

- 训练样本: y ∈ [0, 2560]
- 预测时: `process_frame()` line 310-311 `clamp(0.0, screen_w as f64)` = clamp(0, 1920)
  → 永远落回 [0, 1920], 外侧的 gaze 永远到不了

前端 `SternbergCanvas.svelte:850-851` 也用 `window.innerWidth/innerHeight`,
所以整个数据流都是窗口尺寸, 只有 Rust 这边是 1920x1080。应当:
前端在 `start_screening` 或单独命令里传窗口 size, pipeline 按实际尺寸创建。

### B3. `rect_feature` 计算宽高时未 clamp 最小值 — 可能产生非法 tensor
**文件**: `pipeline.rs:101-116` vs Python `gaze_system.py:332-343`

Python 版本:
```python
lw, lh = max(1, lx2 - lx1), max(1, ly2 - ly1)
```
Rust 版本 (pipeline.rs:105-108):
```rust
(xyxy.2 - xyxy.0) as f32 / w,    // 没有 max(1, ...)
(xyxy.3 - xyxy.1) as f32 / h,
```
虽然 `landmark_rect()` 在 pipeline.rs:76-78 有 `if x1 <= x0 || y1 <= y0 { return None; }` 保护,
但 eye 区域如果被 crop 到图像边缘 (例如头部倾斜到画面边缘), `x0/y0` 在 71-73 行被
`max(0.0) as u32` clamp 之后, 和 `x1` 可能相等再加 1 像素都没有 → `w=0/h=0`, AFFNet rect_feature
第 3/4/7/8 维得到 0, 严重影响 AFFNet 输出。且 `crop_and_preprocess` 在 pipeline.rs:132-133 虽然
`.max(1)` 兜底, 但 rect_feature 本身的值还是 0, 会污染 AFFNet 输入。

### B4. `rect_feature` 和 AFFNet 的眼/脸 rect 定义与 Python 不匹配 (margin 单位换了)
**文件**: `pipeline.rs:34-39, 348-359` vs `gaze_system.py:136-137, 314-320, 623-625`

Python:
```python
LEFT_EYE_INDICES = [362, 385, 387, 263, 373, 380]      # 只 6 个 EAR 点
RIGHT_EYE_INDICES = [33, 160, 158, 133, 153, 144]
def eye_rect_from_landmarks(..., margin=6):            # 像素单位
    x_min -= 6;  x_max += 6
```
Rust:
```rust
LEFT_EYE_INDICES: 16 个点 (包括全眼周轮廓)
RIGHT_EYE_INDICES: 16 个点
landmark_rect(..., margin=0.3)   // ratio 单位: w * 0.3 ≈ 20-30 px
```
两个差异叠加起来 AFFNet 输入的 eye crop **和 Python 版完全不同大小、不同像素**。
如果 AFFNet 是在 Python 管道产生的 112x112 裁切上训练的, Rust 这里送进去
的输入分布和训练分布不一致 → AFFNet 输出的 2D gaze 不可靠 → 8D 特征后两维有偏。

### B5. `start_screening` 允许 pipeline=None 但仍然启动 gaze_loop, 无 gaze 事件发不出
**文件**: `commands.rs:105-126, 169-184`

当 `GazePipeline::new()` 失败 (例如 ONNX 模型缺失):
- state.pipeline = None (line 114 的 fallthrough)
- 但 `camera::CameraCapture::start_with` 仍然启动摄像头 thread (line 118)
- `stream_gaze_loop` 在 line 171 `if let Some(pipe) = ...` 时 `None`, `frame_result = None`
- 所以永远不会 emit `gaze_frame`
- 前端所有 listen('gaze_frame', ...) 永远收不到东西, 既无错误也无进度

**修复**: pipeline=None 时至少 emit 一个错误事件, 或 `start_screening` 返回 Err。

### B6. `finish_screening` 把 fps 写死 30.0, 但 Python 版 fps 可能不同
**文件**: `commands.rs:281` vs camera actually `640x480 @ 30 requested`

```rust
let fps = 30.0;   // line 281
```
如果摄像头不支持 MJPEG 30fps (很多内置摄像头是 YUY2 @ 15-30), `Camera::new` 在
camera.rs:82 用 `RequestedFormatType::Closest` 可能拿到 15 fps。此时:
- `inference::extract_features(&trials, fps=30.0)` 计算 `baseline_frames = fps*0.5 = 15`
- 但实际 pupil_series 里只有 7 个 baseline 样本 → 所有 pupil_* 特征被错误计算

修复: `CameraCapture::start` 应该把 `actual_fmt.frame_rate()` 暴露出来,
`finish_screening` 用实测 fps。当前 camera.rs:102-109 已经读出来了但没透出。

### B7. 校准数据清空缺失 — 前一个 session 的校准残留
**文件**: `commands.rs:75, 232-252`

`start_screening` line 75-77 清空了 `calibration_samples` / `trial_results` / `streaming`, 这一点是好的。但 `finish_calibration()` 没清空 `calibration_samples`, 所以如果用户
再次点 "重新校准", 新旧样本都会进到 Ridge (除非重新 start_screening)。
更严重的是 **`finish_calibration` 里没做 `n < 5` 的早期返回** (有做), 但
如果 features 都是 zero-vector (见 B1), Ridge 拟合时 `self.std` 会在
gaze_math.rs:284 变成 1e-10, 预测时除以 1e-10 会爆炸到 NaN/Inf。
实测 B1 条件下, `pipeline.predict()` 返回值会远超 clamp 范围, clamp 之后
`screen_x/y` 都停在 (0, 0) 或 (1920, 1080)。

### B8. `crop_and_preprocess` 每次都 `rgb.to_vec()` — 每帧多份完整拷贝
**文件**: `pipeline.rs:129, 208` (face_mesh.rs 同样问题)

```rust
ImageBuffer::from_raw(img_w, img_h, rgb.to_vec())   // :129 每次 clone 640x480x3 ≈ 900 KB
```
每帧调用 3 次 (left_eye / right_eye / face), **加上 face_mesh.rs:208** 再 1 次,
共 4 次 × 900KB = 3.6 MB 每帧的无谓分配。30fps 下 108 MB/s 堆压力, 导致 jank 与 GC 抖动。
应改为接受 `&[u8]` 的 shared 视图 (例如把 ImageBuffer 构造挪出去, 在 `process_frame`
开头只做一次)。

## 🟠 逻辑问题 / 可疑设计

### L1. `FeatureBuffer::weighted_mean` 要求 n ≥ 3, 但 `MIN_FEATURES_FOR_PREDICTION = 3`
**文件**: `gaze_math.rs:374-396`, `pipeline.rs:21, 306-307`

两处阈值冗余, 且 buf.len()==3 时 `weighted_mean` 才进入后续逻辑。没问题但两个常量
没有单一真实来源, 改一个忘改另一个就会退化 (pipeline.rs:306 改成 5, mean 还是 3 起返回)。

### L2. `FixationSmoother` 无 reset 时机控制 — 校准完后第一次 predict 会"瞬移"
**文件**: `pipeline.rs:412-414`, `gaze_math.rs:190-193`

`calibrate()` 里 `self.smoother.reset()` 好。但如果用户中途"重新校准" → 再
`process_frame` 第一次输出的坐标是 raw 值 (smoother.x=None), 不会在 saccade 门槛
下限速, 屏幕会出现 1-frame 瞬移。不是 bug 但用户体感差。

### L3. `PupilSmoother` 对 NaN 的处理不完全符合 Python
**文件**: `gaze_math.rs:214-225` vs Python `gaze_system.py:646-658`

Python 语义:
```python
if raw is None or (isnan(raw) or raw == 0.0):
    return raw if raw is not None else float("nan")   # 不把坏样本推进 buf
```
Rust (gaze_math.rs:215-217):
```rust
if raw.is_nan() || raw == 0.0 { return raw; }   // raw==0 时返回 0.0
```
两者都是"跳过 buf", OK。但 Python 返回 NaN (raw is None), Rust 会返回 raw(=0.0)。
对下游 pupil 特征 (`pupil_std`/`pupil_slope`), **0 和 NaN 语义不同**:
NaN 在 inference.rs 应该会被过滤, 0 会被当真实样本参与均值 → 特征被偏移。

### L4. Ridge fit 里 `solve` 失败时 weights 保持 0 但 `fitted=true`
**文件**: `gaze_math.rs:313-322`

```rust
let decomp = xtx_reg.clone().lu();
if let Some(w) = decomp.solve(&xty) {
    for j in 0..d1 { self.weights[j][out_dim] = w[j]; }
}
// 即使 solve 返回 None, fitted=true 仍会在 line 322 被设置
self.fitted = true;
```
当校准数据病态 (所有样本 X 都几乎相同, xtx 接近奇异) 时, `solve()` 可能返回 None,
但 `predict()` line 327 `if !self.fitted { return (0.0, 0.0); }` 通不过 — 返回
全零权重的预测。应当: 任一 out_dim 的 solve 失败就 `fitted=false`。

### L5. `compute_iris_features` 未检查 lm 长度
**文件**: `gaze_math.rs:31-74`

函数直接 `lm[33], lm[133], lm[468], lm[473], lm[1], lm[152], lm[10], lm[234], lm[454]`。
调用处 `pipeline.rs:277-283` 只检查了 `lm.len() >= 468`。`lm[473]` 要求长度 ≥ 474,
`lm[468]` 要求 ≥ 469。如果 face_mesh 返回的 landmark 数正好 = 468 (iris refine
失败 + pad 到 478 的路径没走), 这里会 panic。虽然 face_mesh.rs:219-227 会 pad 到
478, 但万一将来改坏了就直接 crash。建议把 `lm[i]` 全部替换为 `.get(i)?`。

### L6. `face_mesh.rs::detect_face` 裸概率阈值写死 0.5, 且 sigmoid 后比较不过滤已按 0.5 过滤过的 raw
**文件**: `face_mesh.rs:289-297`

```rust
for i in 0..num_anchors.min(scores_flat.len()) {
    let raw = scores_flat[i];
    let score = 1.0 / (1.0 + (-raw).exp());     // ← 每个 anchor 都算一次 sigmoid
    if score > best_score && score > 0.5 { ... }
}
```
等价于找 raw > 0 中的最大值, 但每个 anchor 都算了一次 exp (640 次/帧)。
次要性能损耗。更重要的是没有 NMS — BlazeFace 原本在 2+ 重叠 anchor 都高分时
会合并, 这里只挑最高的一个 anchor, 如果两张脸或误检上升, 框体会飘。

### L7. Iris crop 里"左右眼" corner 顺序与 Python 不一致的副作用
**文件**: `face_mesh.rs:397-416` vs `gaze_system.py:377-379`

Python 注释:
```python
# 右眼（解剖）: 外角33, 内角133, 虹膜中心468
r_ratio = eye_ratio(33, 133, 468)
# 左眼（解剖）: 内角362, 外角263, 虹膜中心473
l_ratio = eye_ratio(362, 263, 473)
```
Rust face_mesh:
```rust
// Right eye: outer corner 33, inner corner 133
if let Some(iris_pts) = self.extract_iris(img, landmarks, 33, 133) { ... }
// Left eye: inner corner 362, outer corner 263
if let Some(iris_pts) = self.extract_iris(img, landmarks, 362, 263) { ... }
```
一致, 好。但 `face_mesh.rs:448-452`:
```rust
let eye_w_px = ((c2.x - c1.x).abs() * iw).max(10.0);
```
左眼 c1=362, c2=263, `c2.x - c1.x` **为正**; 右眼 c1=33, c2=133, `c2.x - c1.x`
**也为正** (都是从左到右)。OK。但 iris_landmarker 模型是 MediaPipe 的,
它对输入图的**镜像敏感**: MediaPipe 通常要求 "左眼需水平翻转后送入" 以保证两眼
共用一个模型权重。当前 `extract_iris` 对两眼都用相同裁切 (没有水平翻转), 如果原生
iris_landmark.onnx 是从 Google 的 .tflite 直接导出的, **左眼的 5 个点位置会是镜像
错误**。需验证 iris_landmark.onnx 是 side-agnostic 还是需要 flip。

### L8. `submit_calibration_sample` 的 padding 逻辑没处理 features.len() > 8
**文件**: `commands.rs:218-221`

```rust
let mut feat = [0.0f64; 8];
for (i, &v) in features.iter().take(8).enumerate() {
    feat[i] = v;
}
```
`take(8)` 正确。当 len < 8 时自动 0 填充, 没问题。但前端其实可能送 3 个值,
后面 5 个是 0 — 结合 B1, 所有 "padding 逻辑" 反而掩盖了上游数据不对的 bug:
应在 `if features.len() < 8 { log::warn!("..."); }`, 或直接拒绝。

### L9. `stream_gaze_loop` 锁 scope 里 `state.camera.as_ref()` 然后再锁一次拿 pipeline — 两次锁之间有竞态
**文件**: `commands.rs:164-179`

```rust
let frame_data = {
    let state = SESSION.lock();
    state.camera.as_ref().and_then(|cam| cam.latest_frame())
};   // ← 锁释放
// 别的线程 (e.g. finish_screening line 277) 可能在这里把 camera 和 pipeline 都 set None
let frame_result = if let Some(cf) = frame_data {
    let mut state = SESSION.lock();         // ← 再锁
    if let Some(pipe) = &mut state.pipeline {
        ...pipe.process_frame(...)
    } else { None }
};
```
当 `finish_screening` 把 pipeline = None 时, 这里 cf 还在手上, 会执行
`else { None }` — 正确 graceful。但 `streaming` 也被设为 false, 下次 loop 开头
的 check 会退出 — OK。不是 bug, 但注释里提到的 "两个锁 scope" 应该写清楚
"pipeline 在此期间可能被释放, 这是预期的"。更健康的模式是 Camera 和 Pipeline
用独立的 Mutex/Arc, 避免整个 Session 一把大锁。

### L10. Camera thread 在 `cam.frame()` 永远 block 时, `stop_flag` 看不到
**文件**: `camera.rs:147-195`

```rust
loop {
    if *stop_flag.lock() { break; }
    match cam.frame() { ... }      // ← 某些驱动会 block 到下一帧, 无 timeout
}
```
如果摄像头被拔掉, nokhwa 的 `cam.frame()` 可能永远阻塞, `stop()` 里 `handle.join()`
(camera.rs:220) 也会永远阻塞 → 应用卡死。建议在 `frame()` 外套 `try_recv` 或
周期性重置超时。(nokhwa 0.10+ 有 `Camera::frame_raw` / 非阻塞 API 可考虑)。

### L11. AFFNet fallback 返回 `[0.0, 0.0]` 会污染 8D 特征
**文件**: `pipeline.rs:339-405`

当 AFFNet 未加载 (常见情况, 因为 affnet.onnx 有外部 data 文件) 或某帧
crop 失败, 第 7/8 维恒为 0。问题:

- 校准时: 所有 13 个 y 不同但 model_gaze 恒为 (0,0), StandardScaler 把 std
  clamp 到 1e-10, `(0 - 0)/1e-10 = 0 / 0` (NaN) 或 0, 回归权重对这两维接近 0, 无害。
- 但一旦 **某些** 帧 AFFNet 成功, **某些** 失败, 校准数据里 model_gaze 分布是
  双模态: 大部分 (0,0), 少数真值, std 计算会拉大, Ridge 对这两维学到大权重,
  预测时 AFFNet 不稳定就让 gaze 剧烈跳变。
- 修复: AFFNet 失败时 **不要** 进 feat_buf, 或使用单独的 flag 强制 fallback 整个帧
  都不用 AFFNet (只用 6D iris)。

### L12. `extract_iris` 对 c2.x < c1.x 时 `eye_w_px = 0` → max(10, ...) 兜底但方向错
**文件**: `face_mesh.rs:450-452`

已在 L7 里提到。左眼 c1=362, c2=263: MediaPipe 约定 362 在鼻侧, 263 在耳侧,
所以 c2.x > c1.x 没问题。但如果头部极度旋转 (yaw > 80°) 导致透视反转, `.abs()`
救了尺寸却没救 crop 方向。iris 模型在这种场景本来就不可靠, 次要。

### L13. `finish_calibration` 返回固定的 mean_error = 12.0
**文件**: `commands.rs:251`

```rust
Ok(12.0)  // 硬编码
```
前端 CalibrationCanvas.svelte:141 `meanError = err` 直接当实际误差显示给用户。
用户会以为"校准精度 12 px", 实际完全不是计算出来的。应当算 predict vs ground truth。

## 🟡 质量 / 小问题

### Q1. `camera.rs:178` fps 日志除以全程 elapsed 而不是窗口
每 300 帧打一次 `fps = frame_count / t_total`, 前 10 秒的 fps 会混入瞬时。
建议滑动窗口 (最近 300 帧 / Δt)。

### Q2. `log::debug!` / `log::warn!` 在热循环里毫无节流
`pipeline.rs:262, 273, 281` 每丢一帧打一行 debug; 长时间遮脸会 30 行/秒地刷。
RELEASE 下 debug! 被优化掉但 warn! 没有; `camera.rs:186, 191` 的 warn! 在
摄像头拔掉时会 30/s 刷屏。加一个 `last_log_at` 节流器。

### Q3. `face_mesh.rs:164-200` `load_tflite_or_onnx` 先后做一次 ONNX 加载 + 失败回退
对每次启动都尝试 tract_onnx 解析 .tflite, 必然失败后再试 .onnx — 两次磁盘 IO。
可直接根据扩展名分派。

### Q4. `image::ImageBuffer::<_, _>::from_raw` 在 face_mesh.rs:208 用 `rgb_frame.to_vec()`
同 B8, 每次 detect 就 clone 完整帧。face_mesh 内部又会生成 resize 副本
(L117 的 128x128, L328 的 192x192, L470 的 64x64 三次)。

### Q5. `pipeline.rs:377-379` rect_feat tensor 构造用 `.unwrap_or_else(|_| ... zeros)`
```rust
let rect_tensor = tract_ndarray::Array2::from_shape_vec([1, 12], rect_feat.to_vec())
    .unwrap_or_else(|_| tract_ndarray::Array2::zeros([1, 12]));
```
`rect_feat` 是 `[f32; 12]`, `to_vec()` 产出 12 个 f32, reshape 到 [1,12] 不会失败。
不过是无害的防御性代码。

### Q6. `gaze_math.rs:254` `fitted=true` 在 fit 结尾无条件置位 (见 L4)
Already reported.

### Q7. `pipeline.rs:185-200` `affnet.onnx.data` 外部权重的加载路径未验证
tract_onnx 加载带外部 data 的模型需要特殊处理 (`read_with_external_data` 之类)。
当前 `tract_onnx::onnx().model_for_path(path)` 对 `external_data` 的支持不确定,
且没有测试覆盖。如果 affnet.onnx 指向 affnet.onnx.data 但 tract 找不到, fit
会失败, 落到 `Ok -> Err -> None` 路径, 8D 特征退化为 6D+(0,0) (即 L11)。

### Q8. `commands.rs:82-85` 用 `current_exe()` 但不处理 symlink
`.unwrap_or_else(|| PathBuf::from("."))` 在 macOS 上 exe 在 `.app/Contents/MacOS/`,
符号链接解析正常, 但 `exe_dir.join("../Resources/models")` 对 .app bundle OK,
`exe_dir.join("../../models")` 对 cargo dev 模式有效; 但没覆盖 `xattr`/notarization
后的路径。最好加 `fs::canonicalize`。

### Q9. `commands.rs:285-291` RF 模型在 `finish_screening` 时才加载
每次 session 结束都 I/O 读 rf_model.json (~50 MB 300 estimators). 应缓存或用 once_cell。

### Q10. `pipeline.rs:248-255` `invalid` 常量每帧都构造 — trivial, 保留。

### Q11. `face_mesh.rs:82-83` 的边界 clamp 用 `u32` 转换截断而非四舍五入
`(rect.x + rect.w).max(0.0).min(iw) as u32).max(x0 + 1)` — `f32 -> u32` 是截断,
当 rect.x = -0.5 时 as u32 先 = 0 (max(0.0) 救了), 但 x1 = x0 + 0.9 时 `as u32` 截到 x0,
然后 `.max(x0 + 1) = x0 + 1`, 实际放大了 0.1 px。不是 bug 但用 `.round() as u32` 更干净。

### Q12. `gaze_math.rs:265` `assert!(n >= 5)` — 用户层错误用 panic 不合适
```rust
pub fn fit(..) {
    let n = x_rows.len();
    assert!(n >= 5, "need at least 5 calibration points");
```
`finish_calibration` 命令层 (commands.rs:237) 已经 check 了 n < 5 并返回 Err,
所以实际走不到。但作为 library 函数, assert 会 abort 整个进程, 应当改为
`Result<(), ...>` 返回。

### Q13. `commands.rs:131-136` gaze-stream 线程没有保存 JoinHandle
```rust
std::thread::Builder::new().name("gaze-stream".into()).spawn(...)
```
`spawn()` 返回的 JoinHandle 被丢弃, 线程变成 detached。`finish_screening` 只能
通过 `streaming=false` 间接让它退出, 但没有 join 能验证确实退出了。如果 stream
thread 在退出前 panic, 没人知道; 下一次 `start_screening` 会创建第二个 thread,
两个 thread 都抢同一个 Session — 之前的 thread 看到 streaming=true (新一轮)
就不退出, 导致 double-emit gaze_frame。

**修复**: 在 SessionState 存 `stream_handle: Option<JoinHandle<()>>`,
`finish_screening` / `start_screening` 开头 `take().map(|h| h.join())`。

### Q14. `commands.rs:336` `stop_game` 是 TODO
未实现。没有游戏正在 run 时调用没副作用, 但 `launch_game` 后再 `stop_game` 不起作用,
前端会以为游戏结束了而后端进程还在。

### Q15. `commands.rs:193-196` `get_session` 返回 `Ok(None)` — TODO
写在生产代码里可能误导调用方认为 "session 不存在", 而不是 "方法未实现"。

### Q16. `log::info!` 在 start_screening 流程中 5+ 次 (commands.rs:99, 108, 121 等)
一次启动就打 5 行 info, 测试时 noise。可以 demote 到 debug!, 只在错误时 warn/error。

## ✅ 亮点

- **`parking_lot::Mutex`** 替代 `std::sync::Mutex`: 无锁投毒问题,
  跨 catch_unwind 也不会污染状态。(`camera.rs:18`, `commands.rs:6`)
- **Camera 初始化同步等待**: `camera.rs:65, 117-136` 用 sync_channel 让主线程
  等待 camera 真正 open 完成, 避免"线程起来了但 device 没拿到" 的竞态,
  这比 Python 的"启动后 probe 一帧"模式更干净。
- **Graceful degradation**: pipeline / AFFNet 失败时有 fallback, 不会整个 session 崩。
  (pipeline.rs:185-200, commands.rs:110-115)
- **Rust borrow 绕过很优雅**: `finish_calibration` 用 `.collect()` 先把数据
  拿出来, 再 `&mut state.pipeline`, 规避了 self-borrow; `stream_gaze_loop`
  拆两个锁 scope (commands.rs:164-179) 逻辑清晰。
- **FixationSmoother 完全移植了 Python saccade 分段**, 含单元测试
  (gaze_math.rs:422-433)。
- **anchor 生成 + 单元测试** (face_mesh.rs:533-550) 防止 anchor 数字漂移。
- **Ridge fit 用 nalgebra LU 分解** 而不是手写高斯消去, 数值稳定性好
  (gaze_math.rs:299-320)。
- **Camera Drop 自动 stop_stream** (camera.rs:232-236), 不会泄漏 USB 资源。
- **`CameraCapture::is_running()`** 封装了 handle.is_finished(), API 干净。
- **Python `_CaptureWorker` override `Thread._stop` ESC 崩溃问题** 在 Rust 版
  完全规避: Rust 没有 Python 那种 hacky 打线程内部变量的能力, Drop + stop_flag
  语义清晰。

## 📊 总体评分

| 维度 | 分数 | 说明 |
| --- | --- | --- |
| 并发安全 | 4/5 | parking_lot + thread 隔离做得好; 扣分在 stream thread 没 join (Q13), 大锁粒度 (L9) |
| 管线正确性 | 2/5 | B1 (校准特征错通道) + B2 (屏幕尺寸写死) + B4 (eye rect 与 Python 不对齐) 任一都能让 gaze 预测不可用 |
| 错误恢复 | 3/5 | 有 fallback 但 pipeline=None 时依然启动 gaze_loop (B5), camera 拔掉可能 deadlock (L10) |
| 性能 | 2/5 | 每帧 4×900KB clone (B8/Q4), 像素级 for 循环填 tensor 3 次 (face_mesh.rs:241-248, 345-352, 471-478, 每次 192×192×3 = 110k 次 index), 30fps 下 CPU 占用很可能爆表 |
| 代码质量 | 3/5 | 模块划分清晰, 注释 OK; 但多处 TODO/hardcode, assert! 在 library 层, unwrap_or_else 兜底掩盖真实问题 |

**综合: 2.8 / 5** — 架构骨架健康但多个 contract mismatch (跨语言特征定义、跨进程屏幕尺寸、前后端 8D 通道) 会让"预测出来的 gaze 不是你以为的 gaze"。在 UI 跑起来**看起来工作** (因为 clamp 让坐标总在屏幕内), 但 Ridge 回归器实际上拟合的是零向量, 后续 ADHD 分类器吃的就是随机特征。
