## A. 特征提取数学正确性（`inference.rs`）

### A.1 致命等价性偏差

1. **skewness 的 bias 开关与训练不一致（中等严重）**  
   - 训练代码 `model/adhd_pupil_classifier/adhd_classifier.py:227` 使用 `stats.skew(valid_rt, nan_policy='omit')`，scipy 默认 **`bias=True`**（即未做样本量校正的 `g1 = m3 / m2**1.5`）。  
   - Python 推理 `inference.py:81` 保持默认 `bias=True`，与训练一致。  
   - Rust `inference.rs:746-755` 用 `g1 * sqrt(n*(n-1))/(n-2)`，即 **`bias=False`**（Fisher-Pearson 校正）。  
   - 结果：Rust 算出的 `rt_skewness` 比训练分布大一个缩放因子（在 n≈160 时约 `1.006×`，数值影响很小；但在稀疏 trials 下偏差会放大）。  
   - **注意**：任务 prompt 里写 "skewness 已对齐 scipy bias=False"，这是把 "与 scipy 对齐" 错写成目标；实际训练用的就是 bias=True，这是方向写反的"修复"。

2. **`rt_diff_correct_incorrect` 空集返回值不一致（轻-中度严重）**  
   - Python `inference.py:91-98`：`rt_correct`/`rt_incorrect` 无样本时 → `nan`，触发 `if np.isnan: rt_diff_ci = 0.0`。  
   - Rust `inference.rs:96`：`mean(&rt_incorrect) - mean(&rt_correct)`，而 `mean(&[])` 返回 `0.0` → 得到 `-mean(rt_correct)` 或 `mean(rt_incorrect)`，非 0。  
   - 末尾的 NaN/Inf 清洗挡不住这个（数值不是 NaN）。  
   - 影响：冷启动或极端 session（全对 / 全错）下，此特征符号和量级均与训练分布不符。

3. **skewness 最小样本阈值差 1（轻微）**  
   - Python `inference.py:81-82`：`len(valid_rt) > 3` → n≥4 才计算。  
   - Rust `inference.rs:747`：`v.len() < 3 return 0` → n≥3 即计算。  
   - 边界上（恰好 n=3）行为不同；训练侧默认门槛是 n>3。

### A.2 稀疏场景最小样本阈值

- `inference.rs:115` `pupil.len() < baseline_frames * 3 → skip`：30 fps 下需 pupil 序列 ≥45 帧（≈1.5 s），与 Python `inference.py:109` 完全一致 ✓。  
- `inference.rs:140` `post_data.len() < 10 → skip`：与 Python `inference.py:131` 一致 ✓。  
- `inference.rs:154-160` `slope_len > 5` 才算 slope，否则 0：与 Python 一致 ✓。  
- `inference.rs:211` load 差分需要 `l1.len() > 2 && l2.len() > 2`：与 Python 一致 ✓。

### A.3 单位修正（Slope）

- `inference.rs:154-157` `slope_len = (post_data.len() / 3).min((fps * 1.0) as usize)`；`slope_raw * (fps / 1000.0)` 将 30 fps 的"Δ%/帧"换回训练尺度下的"Δ%/ms"。与 Python `inference.py:138-145` 完全对齐 ✓。  
- 但 **`pupil_slope_var` 的单位修复并未传递**：  
  - Python 注释 `inference.py:15-18` 明确说 "The same fix is applied to pupil_slope_var for consistency"，但实际上 Python `inference.py:185-186` 直接对已换算的 `slope` 列表做 `np.std`，隐式传递了单位。  
  - Rust `inference.rs:207` 同样 `std_dev(&slopes)`，而 `slopes` 已经是换算后的。  
  - **✓ 没问题**，注释说明文档可以去掉"同样修复"的说法，因为它是自动传递的。

### A.4 Baseline 长度

- `baseline_frames = (fps * 0.5).max(1.0) as usize` → 30 fps 下 15 帧 = 500 ms。训练在 1000 Hz 下 `baseline_samples=500` = 500 ms。**时间维度一致 ✓**。

### A.5 Pupil 基线校正

- `inference.rs:127` `if baseline == 0.0 || baseline.is_nan() { continue; }`：`baseline_frames` 切片是非 NaN 的 `valid_idx` 前缀，mean 理论上不会是 NaN；但若所有 baseline 值全为 0（极少见）会跳过该 trial。Python 同款，OK ✓。  
- `corrected = (pupil - baseline) / baseline.abs()`：Python 用 `abs(baseline)`，Rust 用 `baseline.abs()`。一致 ✓。

### A.6 Gaze 特征

- **最近修复验证（gaze NaN 配对过滤）**：  
  - `inference.rs:234-244` 每 trial 内 `i` 配对遍历，`gx.is_nan() || gy.is_nan()` 任一为 NaN 即丢弃此索引。**✓ 正确的配对过滤**。  
  - Python `inference.py:219-225` 是 `all_gx.extend(t.gaze_x_series)` 再 `valid_g = ~np.isnan(gx) & ~np.isnan(gy)`，等效配对过滤。语义一致 ✓。  
- **`gaze_path_normalized` 的潜在语义问题**：  
  - Python `np.diff(vx)` 只对"保留下来的"有效样本做差分。若中间丢弃了大量 NaN 帧，`diff` 计算的是跨时间间隔的"跳跃距离"，而非真实瞬时位移。  
  - Rust 做同样的事（`i..all_gx.len()` 相邻非 NaN 对），与 Python 行为一致 ✓，但二者都有该问题——**不视为 Rust 实现 bug，只是数据契约风险**。

### A.7 Distractor 梯度

- `inference.rs:308` `valid_d.len() >= 3` 且 `linear_regression_slope(xd, yd_rt)`：与 Python `inference.py:266` 的 `len(vd) >= 3` + `np.polyfit(..., 1)[0]` 一致 ✓。

### A.8 Helper 正确性

- `mean`, `variance`, `std_dev`：空切片返回 0.0、总体（非贝塞尔）方差，与 `np.var / np.std` 默认 `ddof=0` 一致 ✓。
- `linear_slope`：`x = 0..n-1`，闭式解，等价于 `np.polyfit(np.arange(n), y, 1)[0]` ✓。
- `linear_regression_slope`：闭式解 = `np.polyfit(x, y, 1)[0]` ✓。
- `trapezoid`：`s += (v[i-1] + v[i]) / 2.0`，步长视为 1；等价于 `scipy.integrate.trapezoid(y)` 默认 `dx=1` ✓。
- `skewness`：见 A.1，使用 bias=False 形式，与训练的 bias=True 不一致 🔴。

## B. Random Forest 预测（`inference.rs`）

### B.1 决策树遍历（`predict_single_tree` 710-725）

- 使用 `children_left[node] == -1` 判定叶子，与 sklearn 的 TREE_UNDEFINED 语义等价 ✓。  
- `x[feat_idx] <= threshold` 走左，否则右：与 sklearn 完全一致 ✓（sklearn 也是 `<=`）。  
- 返回 `tree.value[0][node].clone()`，形状 `[1][n_nodes][n_classes]` 的提取方式依赖 JSON 中 `value` 嵌套。**未做长度校验，若 JSON 非法（`feature[node] < 0` 在非叶节点）会 panic**：实际 sklearn 导出中非叶节点 feature 均 >= 0，低风险。  
- `children_left[node]` 索引若被污染可 panic，同上。

### B.2 StandardScaler

- `(v - scaler.mean[i]) / scaler.scale[i]`（`inference.rs:445-448`）：正确，与 `sklearn.preprocessing.StandardScaler.transform` 等价 ✓。  
- `scaler.scale` 在 12 维空间上，`x_selected` 也是 12 维（从 27 维 `selected_indices` 抽取）。长度 12 == 12 ✓（已用 `rf_model.json` 验证：`all_features len=27, selected_features=12, scaler.mean len=12, selected_indices=[0,1,2,4,6,8,9,12,13,18,21,22]`）。

### B.3 12 特征抽取

- `x_full` 先按训练顺序 27 维取值（`inference.rs:435-437`），再按 `selected_indices` 抽 12 维（`438-442`）。顺序正确 ✓。

### B.4 概率归一化

- 每树：`votes / votes.sum()` 得到树级概率；  
- 全局：累加树级概率 ÷ `n_estimators` → ensemble 平均。  
- 与 sklearn 的 `predict_proba`（soft voting）**一致 ✓**。  
- `s.max(1e-10)` 避免除零：对零叶子值（理论不可能在 sklearn 中出现，除非 `min_samples_leaf=0` 且无样本到达）做了兜底，OK ✓。

### B.5 Feature Importance（`inference.rs:507-521`）

- **不是 Gini importance**：仅统计每特征在决策节点出现次数 / 总节点数。  
- 与 sklearn 的 `feature_importances_`（按基尼减少量加权）**语义不同**。但代码注释已明确承认"for display purposes this is good enough"。  
- 影响：前端展示的重要性排序与 Python 后端输出可能差异明显，尤其对低基数但高分裂度的特征。如果前端把它作为"科学解释"呈现，会误导用户。  
- **建议**：要么在模型导出 JSON 时加一列 `feature_importances`（sklearn 直接提供），要么在命名/UI 上改成 "split count" 并注明。

### B.6 CI 置信区间（`inference.rs:472-494`）

- 采用经验分位数（最近邻法）：`idx = (p * (len-1)) as usize` 取样本。  
- 对 n=300 的森林：`0.025 * 299 = 7.475 → idx=7`；`0.975 * 299 = 291.525 → idx=291`。  
- `.min(len-1)` 兜底 `usize` 转型不会越界 ✓。  
- 边界 `tree_adhd_probs.is_empty()` 时退回 `(0, adhd_prob, adhd_prob)`（`inference.rs:475-476`），但 `n_estimators = 300` 时这条分支不会触发（除非 JSON 被改坏）。OK ✓。  
- 方差用**总体方差**（除以 n，不是 n-1）：对 n=300 差异可忽略；与 `np.std(..., ddof=0)` 一致。  
- **注意**：经验分位数用最近邻而非线性插值。对 α=0.025/0.975、n=300 差别 < 一个样本，可接受。

## C. 6 维 Attention Profile

### C.1 Sigmoid 校准参数

- 注释 `inference.rs:578-582` 明确来源：`model/adhd_pupil_classifier/features.csv` 上的 22 Control / 28 ADHD 两组中位数。`center = (p50_ctrl + p50_adhd)/2`，`k = 2*ln(4) / |Δp50| ≈ 2.77/Δ`。  
- 几何含义：sigmoid 在两组中位数处取值 ≈ 0.2 / 0.8，合理 ✓。  
- 但 **样本量仅 50 人**（且 2:3 不平衡），中位数不稳健，尤其对长尾特征 `acc_distractor_slope`：`k = 1728.6` 反映 Δp50 ≈ 0.0016 极微小，导致 sigmoid 过度陡峭——**此维度对真实个体会被"压扁"到几乎 0 或 100，丢失区分度**。

### C.2 维度语义与 invert 方向

- `sustained`（持续注意力）：  
  - Python 注释 `// ADHD kids have high skewness from long-tail slow RTs as they lose focus` 与参数 `ctrl_p50=0.6525, adhd_p50=0.4338`——**ADHD 的中位数比 Control 低**！与"ADHD=高 skewness"的直觉相反。  
  - 这是 subagent 之前指出的"ADHD 在某些特征上更低"的语义风险。代码 `sigmoid_map(skew, 0.543..., 12.67, true)` invert=true，即 `raw 低 → 得分高`；而事实上 ADHD skew 更低，所以会得"更高的 sustained_attention"——**评分语义反转**！  
  - 同样 `pupil_late_minus_early` 的 ADHD=1.430 < Control=1.557，invert=true 同样把 ADHD 推高。  
  - **🔴 结果**：对"持续注意力"，ADHD 儿童在此数据集上可能拿到比 Control 更高的分数，与前端展示的"持续注意力=分数越高越好"完全矛盾。  
  - 相同现象需要复核：`cognitive_load_sensitivity`（ctrl `rt_load_diff=57.25 > adhd=39.62` — ADHD 反而受载荷影响更小 → invert=true 会让 ADHD 得分更高）、`gaze_control` 的 `gaze_var_x`（ctrl 6871 > adhd 6074，invert=true → ADHD 得分更高）。这些是样本的方向**与直觉/文献相反**，sigmoid 参数照搬了样本分布但没检查方向学意义。

### C.3 其它维度

- `response_stability`：`cv_rt` ctrl=0.2296 < adhd=0.3092，invert=true → 低 CV 得高分，ADHD 得低分，**方向正确 ✓**。  
- `distractor_resistance`：rt_slope ctrl=11.45 vs adhd=12.46（几乎一样）；acc_slope 差更小（0.029 vs 0.031）。sigmoid k 值巨大（1728.6）→ 对真实受试几乎无分辨力。  
- `pupil_engagement`：max_peak ctrl=8.04 < adhd=10.34，invert=false → ADHD 得分更高。这与"认知投入高"的正面叙事冲突：ADHD 儿童瞳孔峰值高可能是补偿性努力，但在 UI 上被当成"积极参与"。方向学层面可争议。  
- `clamp(0, 100)`：保险，正确 ✓。

### C.4 `AttentionProfile::default()`

- 全部 50.0——**持久化读取报告时（`storage.rs:150`）的默认值**。`get_report` 从 DB 重建时 profile 全部丢失为 50，参见 §D.1。

## D. 存储（`storage.rs`）

### D.1 Schema 严重缺漏

`adhd_reports` 表（`storage.rs:31-41`）只持久化 7 列，以下字段**落盘丢失**：

| 字段 | 代码位置 | 落盘情况 |
|---|---|---|
| `attention_profile`（6 维） | `inference.rs:412-413` | ❌ 未存；`get_report` 时硬塞 `AttentionProfile::default()` → 全 50.0 |
| `block_stats`（block 时序） | `inference.rs:414-415` | ❌ 未存；`get_report` 返回 `Vec::new()` |
| `adhd_probability_std` | `inference.rs:416-417` | ❌ 未存；read-back = 0.0 |
| `adhd_probability_ci95_low` | `inference.rs:418-419` | ❌ 未存；read-back = 0.0 |
| `adhd_probability_ci95_high` | `inference.rs:420-421` | ❌ 未存；read-back = 0.0 |
| 原始 trials（160 条） | `TrialResult` 列表 | ❌ 完全未落盘，无法复跑推理 |
| gaze_frames（30 Hz 流） | pipeline 输出 | ❌ 未落盘 |
| calibration samples | `SESSION.calibration_samples` | ❌ 未落盘 |

**影响**：

1. **历史报告 UI 会退化**：用户点击"查看历史"时，attention_profile 固定为全 50，CI 全为 0，block_stats 为空 → 雷达图/时间线/误差棒全部失真。  
2. **无法做离线重跑 / 模型升级回放**：没存 trials，换模型后历史数据只剩一个聚合数字，失去复现价值。  
3. **`storage.rs:148` `feature_values: serde_json::from_str(&fv).unwrap_or_default()`**：fv/fi 序列化错误会静默返回空 HashMap，吞掉错误。

### D.2 `save_report` 的 `INSERT OR REPLACE`

- `storage.rs:111-124` 用 `INSERT OR REPLACE` 按 `session_id UNIQUE` 唯一约束。  
- **行为**：先 DELETE 同 session_id 的旧行（包括 AUTOINCREMENT 的 id），再 INSERT → `adhd_reports.id` 会跳号，但用户不感知。  
- **风险**：如果后续加外键引用 `adhd_reports.id`（比如 attention_profile 表的 report_id），REPLACE 会把旧外键级联删除，外键表孤立。现阶段无外键依赖，暂时安全，但扩展 schema 时要改成 `ON CONFLICT DO UPDATE`。

### D.3 会话状态机不完整

- `sessions.status` 定义了 `pending`（`storage.rs:27` 建表默认）。  
- `save_report` 在同一事务（其实是两条独立 execute）里 UPDATE status = 'completed'（`storage.rs:126-129`）。  
- **但**：  
  - 没有 `running` 状态写入：`start_screening` 只创建 pending，pipeline 启动后未更新为 running。  
  - 没有 `failed` / `cancelled` 路径：若 `finish_screening` 报错（模型加载失败等），session 永远停在 pending。  
  - 没有 `stop_game` 或其它"用户取消"的 hook 更新 status。  
- **结果**：数据库里会累积大量 pending 的死会话；前端 `list_subject_sessions` 展示会混乱。

### D.4 并发

- `static DB: OnceCell<Mutex<Connection>>`（`storage.rs:11`）：单连接 + Mutex。  
- 所有查询串行化，每个命令加锁—读/写—解锁。rusqlite 默认连接是单线程安全的，这是正确的保守做法；但若以后有高频 gaze_frames 批量写入（30 fps × 多 trial），这单锁会成瓶颈。  
- `init_db` 使用 `DB.set(...).map_err(|_| "DB already initialized")`（`storage.rs:43`）：幂等性保护 ✓。

### D.5 路径处理

- `lib.rs:27-35` 使用 `app.path().app_data_dir()`——Tauri v2 官方 API，跨平台（macOS `~/Library/Application Support/<bundle-id>`, Windows `%APPDATA%/<bundle-id>`, Linux `~/.local/share/<bundle-id>`）。  
- `create_dir_all(&app_dir).ok()`：忽略错误（已存在或真失败都吃掉）。若目录真的创建失败，下面 `init_db` 会崩，但错误信息会含 path，可诊断。可以接受。

### D.6 `list_subject_sessions` 排序

- `ORDER BY started_at DESC`（`storage.rs:168`）。`started_at` 默认 `datetime('now')` 精度到秒；同一秒内两条 session **排序不稳定**。session_id 是 UUID，无单调性。建议 `ORDER BY started_at DESC, id DESC` 做 tiebreak，或改用 `INTEGER PRIMARY KEY` 的自增列。

### D.7 子表缺失

表中**没有** `trials`, `gaze_frames`, `calibrations`, `attention_profile`, `block_stats`, `uncertainty` 这些子表。完整的会话数据模型（和 Python 版 `storage/models.py` 对齐）缺失。

### E.3 错误类型

- 所有返回 `Result<T, String>`。字符串错误给前端展示时，中文 fallback 由前端补；包含底层路径信息（如 `Failed to load RF model from ".../models/rf_model.json"`）会泄露用户文件路径——可考虑脱敏。

### E.4 参数命名

- Tauri v2 默认 **snake_case 自动转 camelCase** 于 JS 侧；Rust 侧保持 snake_case 即可。`start_screening(subject_id: String)` 前端 invoke 要传 `{ subjectId }`——需前端约定。扫描前端代码验证略。

### E.5 SESSION 全局锁

- `static SESSION: Lazy<Mutex<SessionState>>`（`commands.rs:31-41`）用 `parking_lot::Mutex`。  
- 每个命令先 `SESSION.lock()`——**所有命令相互串行**。典型场景：`submit_calibration_sample` 被校准点击触发（快速多次），`submit_trial_result` 每 trial 结束触发——高频写入串行化在此单锁上，配合 30 fps gaze_stream 也抢锁。  
- `stream_gaze_loop`（`commands.rs:142-191`）为了避免借用冲突，**两次加锁**（第一次只读 camera，第二次 mut pipeline），中间 drop lock：这个写法绕开了借用检查，但**存在竞态**：两次 lock 之间另一个命令可能把 `state.pipeline` 清掉（如 `finish_screening` 的 `state.pipeline = None`，`commands.rs:278`）——再进第二段就打不到 pipeline，静默丢帧。不算崩溃，但行为非预期。

### E.6 Panic 传播

- Rust panic 在 `#[tauri::command]` 中会变成 Tauri 内部的 `Error::Runtime(InvokeError::...)`，前端 `invoke()` Promise reject，但**错误信息通常是 "internal error"，不含 Rust panic 消息**——用户看不到实际错误。  
- `storage.rs:70` 等处使用 `.unwrap()` 对 Mutex poison。一旦任一 handler panic 且持锁，Mutex 中毒 → 所有后续命令也 panic。  
- 建议改成 `lock().map_err(|e| e.to_string())` 显式处理。

### E.7 `trial_results` 与 `trials` 参数的二义

- `submit_trial_result`（`commands.rs:256-264`）把 trial 追加到 `state.trial_results`。  
- `finish_screening`（`commands.rs:268-318`）**不使用 state.trial_results**，而是直接用参数 `trials: Vec<TrialResult>`。  
- 结果：`state.trial_results` 是彻底的 **dead code**，只是占内存。更严重：如果前端设计上想让 submit_trial_result 作为"持久化"（即中途崩溃也能恢复），当前根本做不到——trials 只在前端内存。  
- **建议**：要么 `finish_screening` 改为 `state.trial_results.drain(..)`，要么删掉 `submit_trial_result`。当前是最坏组合——暴露了接口但不是 source of truth。

### E.8 退出时的流式线程

- `stream_gaze_loop` 依靠 `state.streaming` 标志位退出（`commands.rs:154-158`）。  
- App 退出时 Tauri 会 `std::process::exit`，工作线程**被强杀**，OpenCV/camera 可能不及释放——macOS 下偶发摄像头灯留亮。建议在 Tauri `on_window_event::CloseRequested` 中先设 `streaming=false` 并 join 线程。  
- 目前 `commands.rs:131-136` 线程用 `Builder::name().spawn(...)` 但**没保存 JoinHandle**，无法 join——永远无法"优雅"退出。

## G. 数据生命周期复盘

### 理想流程

1. `start_screening` → SESSION reset、pipeline 构建、camera 启动、gaze 流线程 spawn。session 行写入 pending。  
2. 13 点校准：`submit_calibration_sample` 累加到 `state.calibration_samples`。  
3. `finish_calibration` → Ridge 拟合。  
4. 160 trials：`submit_trial_result` **累加（dead）**；前端也把 gaze/pupil 累加在自己内存里。  
5. `finish_screening(session_id, trials)` → 停流、释放 pipeline、特征提取、RF 预测、写 DB、返回 AdhdReport。

### 失败路径的状态清理

| 失败点 | 内存状态 | DB 状态 | 再次 start_screening |
|---|---|---|---|
| 校准中崩溃 | `calibration_samples` 残留 | session=pending 孤立 | ✓ L76 `.clear()` 重置内存 |
| 校准拟合错误（pipeline None） | Ridge 未拟合 | pending | ✓ 重新开始可覆盖 |
| trial 中用户关 App | state、streaming=true | pending | **❌** 流线程无 join，可能仍在跑 |
| finish_screening RF 加载失败 | pipeline 已 drop | pending 永久卡住 | ✓ 内存 reset OK 但 DB 里堆死行 |
| save_report 失败 | 报告数据丢失 | session 永久 pending | ✓ 同上 |

### `state.trial_results` vs `trials` 参数

- **真实数据源是前端传入 finish_screening 的 `trials` 参数**。  
- `state.trial_results` 只是一份不被读取的副本——对中途崩溃恢复毫无用处。**建议**：要么删 submit_trial_result，要么 finish_screening 从 state 读取并忽略参数。

### 退出 App 时的流线程

- 见 §E.8：无 JoinHandle，无 graceful shutdown hook。

## H. 代码质量

### 优点

- 文件头注释清晰引用 Python 行号，审查友好。  
- 内置单元测试 `test_slope_unit_fix` 覆盖了关键修复；`test_rf_prediction_from_json` 做端到端预测 + CI 形状断言。  
- Rust 借用检查下的两次 lock 绕行写法（`stream_gaze_loop`）有明确注释说明意图，阅读容易。  
- `AdhdReport` 的 Serialize 实现直接支持前端 JSON 反序列化，无需中间 DTO。

### 问题

- `inference.rs:177-182`：`PupilFeats` 手动复制两次（一次塞 trial_pupil_feats、一次塞 pupil_by_load），用 `Clone` derive 或直接引用能消掉。  
- `commands.rs:20` `trial_results` 字段定义但未用——dead code。  
- `commands.rs:281` `fps = 30.0` 硬编码，应与 pipeline 的实际 fps 一致；如果未来换 60 fps 摄像头会静默错误。  
- `storage.rs:43` `.map_err(|_| anyhow::anyhow!("DB already initialized"))` 会掩盖真实 mutex 错误——实际上 `OnceCell::set` 的唯一失败就是"已初始化"，OK，但注释说明一下更好。  
- 注释/实现漂移：`inference.rs:54-56` 说 "slope_len = min(len/3, fps * 1.0)"，实际是 `(post_data.len() / 3).min((fps * 1.0) as usize)`——两个 `min()` 语义一致，但注释用数学符号 `min(a,b)`，读者第一眼可能疑惑 `/3` 的优先级。  
- `attention_profile` 模块里重复的 `sigmoid_map(raw, center, k, true)` 八处调用：可以抽成具名函数（如 `lower_is_better`）提高可读性。

## Top 3 🔴 Bugs 与总结

### 10 行汇报

1. 🔴 **Attention profile 方向学反转**（inference.rs:584-596 等）：sustained_attention / cognitive_load_sensitivity / gaze_control 三个维度的 sigmoid 参数来自 50 人样本中位数，但样本中"ADHD 在这些特征上 = Control 反而相反方向"，而代码默认 invert=true，导致 ADHD 儿童在 UI 上得分可能**更高**——与前端"分数越高越好"叙事直接冲突。
2. 🔴 **SQLite schema 严重缺漏**（storage.rs:31-41）：attention_profile（6 维）、block_stats、CI 上下界 + std、原始 trials/gaze_frames/calibration 全部不落盘；get_report 读回的历史报告 attention_profile 全 50、CI 全 0、block_stats 空——历史详情页严重失真，模型升级也无法回放。
3. 🔴 **Rust skewness 用 bias=False，与训练的 bias=True 失配**（inference.rs:746-755）：训练侧 `stats.skew` 默认 bias=True，但 Rust 加了 `sqrt(n(n-1))/(n-2)` 校正因子，使 `rt_skewness` 值进入已标准化的 pipeline 时偏离训练分布。注释写"对齐 scipy bias=False" 方向写反。另外 `rt_diff_correct_incorrect` 空集不返回 0 与 Python 不一致。

**次严重**：`state.trial_results` 与 `finish_screening(trials)` 并存造成数据源二义（commands.rs:20, 256, 268）；session 状态机只有 `completed`，无 `failed/cancelled/running`，导致崩溃/取消后会话永久卡 pending；gaze 流线程无 JoinHandle、App 退出不优雅；feature_importance 用 split count 代替 Gini（UI 语义可能误导）。

**总体评分 3.0 / 5**：推理数值层面基本正确且与 Python 等价率高，最近两个修复（slope 单位、gaze NaN 配对）到位；但存储模型严重不完整、attention profile 有系统性语义反转风险、Rust/Python 有两处隐蔽数值差。建议优先修复：(a) 持久化完整 AdhdReport 字段；(b) 核查并修正 attention profile 的方向与 sigmoid 陡度；(c) 对齐 skewness 的 bias 设置。
